Framework Choice

Given: static nonprofit site, Google Sheets data source, Mailchimp integration, bilingual with different page sets per language

A

Stay with Astro

What you already have. Static-first, zero JS by default, content collections, built-in i18n support.

Pros

  • Existing work — layouts, components, theming, and i18n infrastructure are already built
  • Zero JS shipped — fastest possible page loads, great for BCS internet speeds
  • Content collections — built-in schema validation for scholar data from Sheets
  • Simple mental model — each .astro file = one page, no framework abstractions
  • Perfect for static sites — this is literally what Astro was built for
  • Netlify deploys just work — already configured

Cons

  • Astro templating syntax is its own thing (not React/JSX, though it's similar)
  • Smaller ecosystem than React/Next if you need UI libraries
  • If you ever need dynamic features (auth, dashboards), Astro is awkward
B

Next.js (React)

Full-stack React framework. Static export mode or server-rendered. Huge ecosystem.

Pros

  • React skills transfer — if you know React, everything is familiar
  • Huge ecosystem — component libraries, tutorials, hiring pool
  • Flexible — can add dynamic features (API routes, auth) later without switching frameworks
  • Built-in i18n — mature internationalization support

Cons

  • Overkill — ships React runtime (~40-80KB) for a site that doesn't need interactivity
  • Rebuild from scratch — nothing from the current Astro codebase transfers
  • Complexity tax — SSR/SSG/ISR modes, App Router vs Pages Router, hydration issues
  • Vercel push — best experience is on Vercel, Netlify deploys work but less smooth
  • Slower page loads than Astro for a static content site
C

Plain HTML + Vite (or 11ty)

No framework at all. HTML templates, CSS, and a build tool for bundling. Maximum simplicity.

Pros

  • Zero abstraction — what you write is what ships
  • No framework to learn or maintain
  • Fastest possible builds
  • No dependencies to update or break

Cons

  • No components — header/footer changes mean updating every HTML file
  • No content collections — you'd build your own Sheet→HTML pipeline from scratch
  • i18n is manual — no framework help for bilingual routing
  • Rebuild everything — none of the current work transfers
  • More code to maintain long-term, not less

💡 The real question

This site has no user authentication, no dynamic data, no forms that write to a database, and no client-side interactivity beyond filtering and toggles. It's a brochure site with a Google Sheet as a data source.

The features you need — bilingual routing, content from external data, static pages, component reuse — are exactly Astro's sweet spot. Switching to Next.js or React buys you ecosystem familiarity but costs you a full rebuild and ships unnecessary JavaScript to users on potentially slow BCS connections.

Recommendation: Stay with Astro. You already have working infrastructure. The new requirements (Sheets integration, different page sets per language, newsletter page) are additions to what exists, not reasons to start over.
