CMS Options for BBF

Who updates what, and how hard should it be?

The non-technical admin needs to update: stats/numbers (yearly), scholar profiles (ongoing), and upload newsletter PDFs (quarterly). Jose handles everything else.

A

No CMS — Google Sheets + Simple Upload

Scholars & stats live in a Google Sheet. Client edits the sheet (they already know how). A build hook pulls data on deploy. Newsletter PDFs uploaded via a simple drag-and-drop form.

Pros

  • Zero new logins — client uses tools they already know
  • Free forever, no vendor lock-in
  • Site stays 100% static (fast, secure, cheap)
  • Jose: full control via code, no CMS to maintain

Cons

  • Changes aren't instant — need a rebuild trigger
  • No preview before publish
  • Newsletter upload needs a small backend (or third-party service)
  • Spreadsheet can be accidentally broken by the client
B

Headless CMS (Decap / Tina / Keystatic)

A lightweight CMS that edits markdown files in your Git repo. Client gets a simple web form with fields for each scholar. Site remains static Astro.

Pros

  • Client gets a friendly form UI — no spreadsheet accidents
  • Changes commit to Git automatically (audit trail)
  • Free tier covers BBF's needs (Decap, Keystatic are free)
  • Jose: still edits code directly when needed
  • Can preview changes before publishing

Cons

  • One more login for the client (GitHub or Netlify Identity)
  • Setup and maintenance overhead for Jose
  • CMS UI can still confuse a non-technical user
  • Vendor dependency (if CMS project dies, need to migrate)
C

Keep WordPress as Headless CMS

Keep the existing WP install purely as a data backend. Astro pulls scholars/stats/newsletters from the WP REST API at build time. Client uses WP admin (or doesn't — you do).

Pros

  • Existing subscriber data and content stays in place
  • WP admin is familiar to many people
  • Mature ecosystem — plugins for anything
  • Can extract newsletter subscribers and Friends of BBF data

Cons

  • Must keep WP hosted and updated (security patches, PHP)
  • WP admin is NOT simple for a zero-tech user — it's overwhelming
  • Hosting cost for a CMS nobody is really using
  • Nobody uses it now — unlikely to start
D

Hybrid — Sheets for Client + Git for Jose

Stats and scholar data in a Google Sheet (client's domain). Jose edits everything else in code. Newsletter PDFs stored in a Google Drive folder that auto-syncs. Best of A without the upload backend problem.

Pros

  • Client only uses Google (Sheets + Drive) — truly zero learning curve
  • Jose keeps full code control, no CMS overhead
  • Free, reliable, no hosting needed for the CMS layer
  • Google Sheets has built-in validation (dropdowns, data types)

Cons

  • Need a build script to pull Sheet data → markdown
  • Still need a deploy trigger (Netlify webhook or manual)
  • Google API setup is a one-time hassle
  • Drive PDFs need a public sharing link or API access

💡 Regardless of choice: extract WP data first

Before killing WordPress, we should extract: newsletter subscribers (emails), Friends of BBF list, and any scholar data that's in WP. This is a one-time migration task that applies to all options above.
